Developer documentation

How the plugin is structured, what it depends on, and where to change behavior.

Requirements

  • WordPress 5.0+
  • PHP 7.2+ (see plugin header in rest-api-route-tester.php for the authoritative minimum)
  • jQuery (bundled with WordPress admin)
  • User with manage_options to use the UI and execute tests

Repository layout

rest-api-route-tester/
├── rest-api-route-tester.php   # Main plugin class, hooks, AJAX handlers
├── assets/
│   ├── app.js                  # Admin UI (jQuery)
│   └── style.css               # Admin styles
├── templates/
│   └── admin-page.php          # Admin screen markup
├── languages/                  # Translations
└── docs/
    └── site/                   # This static documentation site (HTML/CSS/JS)

Runtime flow

  1. On admin_enqueue_scripts, scripts load only on the plugin’s admin page.
  2. wp_localize_script exposes wprrt_vars with ajax_url and nonce.
  3. The UI calls admin-ajax.php actions prefixed with wprrt_ (see Admin-Ajax API).
  4. Tests use rest_do_request() against the active REST server—no separate HTTP client.

Customizing the admin UI

Edit assets/app.js and assets/style.css. Rebuild steps are not required unless you add a separate front-end toolchain.

If you add a bundler later, keep built files in assets/ and enqueue them from PHP so WordPress loads a single script handle.

This documentation site

Plain HTML, CSS, and JS under docs/site/. Replace assets/css/main.css or swap the markup in each page to retheme. Shared chrome is duplicated across pages on purpose so you can move to any templating system later without fighting a build step.
